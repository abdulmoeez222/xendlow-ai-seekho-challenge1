# Implementation Plan: E-Commerce Operations AI Copilot

This plan details the technical architectural changes required to pivot the Insight Engine into an **AI Autonomous Operations Copilot for Shopify/E-Commerce Stores**, fully running on structured mock database schemas with a **Human-in-the-Loop (HITL) Approval Gate**.

---

## User Review Required

> [!IMPORTANT]
> **1. Human-in-the-Loop (HITL) Pipeline Interruption:**
> To implement "Accept/Reject", we must transition the 5-Agent pipeline from a single synchronous request into a **two-phase asynchronous execution**:
> * **Phase 1 (Analysis & Proposal):** The Ingestor, Analyst, and Planner execute, saving candidate decisions in Supabase, and setting the pipeline status to `pending_approval`. The server immediately returns this state.
> * **Phase 2 (Approval & Execution):** The pipeline halts. The user views the proposal on their Flutter App or React Dashboard. 
>   * If **Approved**: The client posts to `/approve-plan`, which triggers the Executor and Reporter agents to run and update mock inventory/ad/pricing tables.
>   * If **Rejected**: The client posts to `/reject-plan`, which either cancels the run or triggers the Planner's predefined `fallback_action`.

> [!IMPORTANT]
> **2. Admin Authentication (1-User Login & Sign Out):**
> To protect the business operations console, we will implement a clean, high-fidelity **Admin Authentication Gate** for a single Admin user (`admin@insightai.com` / `admin123` or pre-registered via Supabase).
> * The Flutter and React apps will open to a gorgeous dark-themed glassmorphic **Login Screen**.
> * Once authenticated, the user session is preserved (saved locally in SharedPreferences for Flutter, and localStorage/Context for React).
> * A **Sign Out** button in the dashboard allows logging out and returns the user securely to the login screen.

## Token & Performance Optimization Plan

To keep API token usage to an absolute minimum and ensure sub-second response times, we will implement the following production-grade optimization techniques:

### 1. Selective Context Retrieval (No Database Dumping)
* **What we avoid:** We never dump the entire products table, competitors catalog, or ad records into the LLM prompts.
* **What we do:** We use Python to pre-filter and retrieve only the **at-risk resources** before sending them to Gemini:
  * For *Logistics/Fuel Shocks*: We only fetch courier rates and low-margin products.
  * For *Marketing/Ad Shocks*: We only fetch ad campaigns with ROAS below the target threshold (< 2.0x).
  * This reduces our input context size by **80% to 90%** (keeping prompts under 1,000 tokens).

### 2. High-Compression Episodic Memory
* **What we avoid:** We never inject full raw historical JSONs of previous runs into the prompts.
* **What we do:** We use a highly compressed, single-line text summary representation for past memories:
  * *Compressed format:* `[Run #10] Scenario: Sales Drop | Action: Campaign (15% Lahore) | Status: APPROVED | Outcome: Successful (PKR 2.4M)`.
  * This condenses what would be a 5,000-token historical record down to just **25 tokens**.

### 3. Strict JSON Constraints (No Output Padding)
* We enforce rigid JSON-only constraints with low temperatures (e.g. `0.2`) on Gemini.
* We instruct the model to skip markdown code fences (```json), conversation fillers, or explanations. This minimizes the output tokens generated, saving cost and improving speed.

### 4. Hybrid Local Constraints (Pre-filtering)
* We execute basic business rule violations (like overstocking or vendor delays) locally in Python code *before* invoking the LLM. If a rule is violated, Python flags it directly to the model as `supplier_delay: true`, saving the LLM from parsing massive raw supplier logs to discover the delay itself.
